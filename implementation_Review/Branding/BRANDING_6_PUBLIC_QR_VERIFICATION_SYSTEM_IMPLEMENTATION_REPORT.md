## Executive Summary

BRANDING.6 implements the **Public QR Verification System** for formal ERP outputs generated through the Executive Ledger engine (BRANDING.5).

The system allows an external party to scan a QR code printed on a formal document (HR letter, certificate, etc.) and verify its authenticity on a mobile-friendly public page — **without login**.

**Key security guarantees enforced:**
- No internal DB IDs in any public URL or response
- Verification tokens are 256-bit cryptographically random (base64url)
- Public lookup uses a `SECURITY DEFINER` RPC — no broad anonymous table access
- RLS enabled and forced on new table
- All public payloads pass strict allowlist sanitization
- No salary, IBAN, medical, or HR restricted data in any public response
- No external QR services — local `qrcode` package only

### 2. Public Verification Library
**Directory:** `src/lib/public-verification/`

| File | Purpose |
|------|---------|
| `types.ts` | DB row types, input types, `PublicVerificationResult`, `SAFE_SUMMARY_FIELDS` allowlist |
| `token.ts` | `generateVerificationToken()` (crypto.randomBytes(32).toString('base64url')), `buildVerificationUrl()`, `buildVerificationPath()` |
| `sanitizer.ts` | `sanitizePublicPayload()` — allowlist + deny-list sanitization; `buildVerificationSummary()` |
| `qr.ts` | `generateQrDataUrl(url)` — local `qrcode` npm package (PNG, 120px, ERP primary colors); `isValidQrDataUrl()` — validates data URL is trusted PNG/SVG only |
| `index.ts` | Barrel export |

**Package installed:** `qrcode` + `@types/qrcode` (21 packages, local generation only)

**Token specs:**
- `crypto.randomBytes(32)` → 256-bit entropy
- `base64url` encoding → 43 URL-safe characters
- Satisfies DB `length >= 32` constraint
- Never includes sequential IDs or report data

**Sanitizer blocked patterns:** salary, wage, payroll, iban, bank_account, account_number, swift, passport, eid, national_id, medical, health, insurance, disciplinary, audit, internal, storage_path, file_path, service_role, api_key, secret, token (except token_label), embedding, vector, ocr_text, extracted_text, raw_payload, prompt, `^id$`, `_id$`

### 5. Executive Ledger Integration
**Updated files:**

- `src/lib/executive-ledger/html-renderer.ts` — Updated QR rendering:

  - If only `verification.publicUrl`, renders text URL fallback
  - If `qrPlaceholder: true` (legacy), renders placeholder box

**Security in renderer:**
- `isValidQrDataUrl()` enforces only `data:image/png;base64,` or `data:image/svg+xml;base64,` URLs
- Prevents arbitrary data URLs from user input
- `publicUrl` must start with `https://` to pass `elEscapeAttr()` URL validation

## Known Limitations / Deferred to BRANDING.7+

1. **Download delivery** — `download_enabled` is supported in DB and public page, but actual file serving is deferred. The page shows "Download not available" placeholder.
2. **Full module-wide rollout** — Only `LetterPreviewDialog` (HR module) has pilot integration. Module-wide issuance is BRANDING.8.
3. **Template governance** — No template versioning or approval workflow. BRANDING.7.
4. **Background expiry job** — Links auto-expire on access (via RPC), but no background cron job to bulk-expire. The RPC handles this lazily on read.
5. **QR in other output types** — Only `LetterPreviewDialog` has the "Issue QR" button. Other output generators (report runner, PDF export) will be wired in BRANDING.8.
6. **Supersede workflow in UI** — `supersedeOutputPublicLink()` server action is implemented but not yet wired to admin UI. Admin can cancel; superseding requires code-level call or BRANDING.8.
