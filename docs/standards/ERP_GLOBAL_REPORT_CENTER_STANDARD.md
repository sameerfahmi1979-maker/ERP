# ERP Global Report Center Standard

## Phase

```
REPORT.2 — Global Report Engine + Registry + Security Foundation
Date: 2026-06-19
```

---

## 1. All ERP Reports Must Use the Global Report Center Engine

Every report, export, letter, certificate, or form generated by any ERP module **must** go through:

```
src/lib/report-center/report-runner.ts → runReport()
src/server/actions/reports/runner.ts   → runReportAction()
```

Do **not** build module-specific report systems outside the global engine.

---

## 2. No Hardcoded Company Names, Logos, or Identity

The report engine supports **unlimited future companies** through configuration.

**NEVER hardcode:**
- Company names (ALGT, ALS, Alliance Gulf, or any other)
- Logo URLs
- Colors
- Footer text
- Signatories

All branding **must** resolve from:
```
erp_report_branding_profiles
→ loaded dynamically by branding-resolver.ts
→ seeded dynamically from owner_companies
```

---

## 3. Branding Resolution Rules

```text
1. If templateId provided → load that template + branding profile
2. If single ownerCompanyId → load that company's default template/branding
3. If multiple ownerCompanyIds → require manual template selection
4. If no ownerCompanyId → use NEUTRAL_DEFAULT / GROUP_DEFAULT fallback
```

Implemented in: `src/lib/report-center/branding-resolver.ts`

---

## 4. Mixed-Company Reports Require Template Selection

If a report data set spans **more than one owner company**, the system returns:
```
requiresManualTemplateSelection: true
```

The UI must prompt the user to select a template before generating the report.
This is non-negotiable — it prevents cross-company branding contamination.

---

## 5. Database Standard

All report center tables use:
```sql
id BIGINT GENERATED ALWAYS AS IDENTITY PRIMARY KEY
```

UUID, SERIAL, and BIGSERIAL are **not allowed** for any report center table.

All FKs to report center tables and from report center tables to ERP tables must be `BIGINT`.

---

## 6. Sensitive Data Redaction

All sensitive data must be redacted **server-side** before it reaches the output adapter.

The double-layer redaction flow:
```
1. Report fetcher checks permissions before building the query
2. applyRedaction() strips/masks any remaining sensitive fields per the report's sensitive_profile
3. Only the sanitized ReportDataResult is returned
```

**Never return raw sensitive values (salary, IBAN, medical data, etc.) to the client.**

Profiles:
```
normal | payroll | medical | disciplinary | recruitment | dms_confidential | mixed_sensitive
```

Implemented in: `src/lib/report-center/redaction-engine.ts`

---

## 7. Report Run History Stores Metadata Only

`erp_report_runs` stores run metadata: who ran it, when, output format, row count, redaction summary.

**Never store raw report data rows in `erp_report_runs`.**

---

## 8. HR.11 Is a Consumer of the Engine

HR.11 (HR Reports Library) will add HR-specific fetchers to `REPORT_FETCHERS` in `report-fetchers.ts`.

HR.11 must **not** build a parallel reporting system. All HR reports go through the global runner.

---

## 9. Permissions

| Permission | Purpose |
|---|---|
| `reports.view` | View registry, templates, branding profiles |
| `reports.run` | Generate reports, create run logs |
| `reports.export` | Export to PDF, Excel, CSV, print |
| `reports.email` | Send reports via email, create delivery logs |
| `reports.manage` | Create/update templates, branding, registry entries |
| `reports.history.view` | View all report run history |
| `reports.sign` | Include stamp/signature in output (REPORT.3+) |

---

## 10. Query Keys

```typescript
queryKeys.reports.registry()
queryKeys.reports.registryDetail(reportCode)
queryKeys.reports.templates()
queryKeys.reports.brandingProfiles()
queryKeys.reports.run(runId)
```

---

## 11. Adding a New Report

1. Insert a row into `erp_report_registry` (via migration or admin action).
2. Implement a `ReportFetcher` in `src/lib/report-center/report-fetchers.ts`.
3. Register it in `REPORT_FETCHERS`.
4. Assign appropriate `required_permissions` and `sensitive_profile`.
5. The branding resolver and redaction engine apply automatically.

## REPORT.3 Additional Rules (Output Adapter Engine)

Added in phase REPORT.3 (2026-06-19):

1. **All output adapters must accept `ExportBrandingContext`** (optional `branding` field on `ERPExportOptions`).
2. **Hardcoded company footer text is forbidden.** Use `branding.footerTextEn ?? "ERP Report"` neutral fallback.
3. **Mixed-company output must block until template selected.** When `requiresTemplateSelection=true`, `ERPExportMenu` calls `onRequireTemplateSelection` instead of proceeding.
4. **New owner companies must receive report branding/template defaults.** Call `ensureReportBrandingForOwnerCompany(id)` after company creation.
5. **Existing exports without branding continue to work unchanged** — `branding` field is optional.
6. **Template designer is not required.** Template management UI (list/edit) is sufficient for this phase.
7. **`ResolvedReportTemplate` is server-only.** Pass `ExportBrandingContext` (serializable flat struct) to client components.
8. **`ReportTemplateSelectDialog`** must be shown for multi-company or manual-required strategy before output.
9. **`resolveTemplatePreview(templateId)`** is the canonical way to convert a template to `ExportBrandingContext`.
