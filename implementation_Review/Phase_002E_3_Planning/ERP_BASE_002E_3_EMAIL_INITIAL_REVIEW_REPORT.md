- `src/server/actions/roles.ts`

**Current Pattern**:

```typescript
// Server action pattern (consistent across all modules)
export async function createOrganization(input: CreateOrganizationInput) {
  "use server";
  
  try {
    // 1. Get auth context
    const ctx = await getAuthContext();
    
    // 2. Check permissions
    if (!hasPermission(ctx, "organizations.manage")) {
      return { success: false, error: "Permission denied" };
    }
    
    // 3. Validate input (Zod)
    const result = schema.safeParse(input);
    if (!result.success) {
      return { success: false, error: formatZodErrors(result.error) };
    }
    
    // 4. Database operation (Supabase client)
    const supabase = await createClient();
    const { data, error } = await supabase.from("owner_companies").insert(...);
    
    // 5. Log audit
    await logAudit({
      module_code: "organizations",
      entity_name: "owner_companies",
      entity_id: data.id,
      action: "create",
      new_values: data,
    });
    
    // 6. Revalidate and return
    revalidatePath("/admin/organizations");
    return { success: true, data };
  } catch (error) {
    return { success: false, error: String(error) };
  }
}
```

**Key Characteristics**:

✅ `"use server"` directive  
✅ Consistent error handling  
✅ Permission checking via RBAC  
✅ Zod validation  
✅ Audit logging (success only currently)  
✅ Path revalidation for data refresh  
✅ Typed returns (`{ success, data?, error? }`)

## 📐 Architecture Decisions

### Decision 1: Client-Side vs Server-Side Attachment Generation

**Options**:

**A) Client-Side Generation** (like current download export):
- Generate attachment in browser (PDF/Excel/CSV)
- Convert to base64
- Send base64 to server action
- Server action forwards to Microsoft Graph

**Pros**: Reuses existing export functions  
**Cons**: Large base64 data in request body (10MB+ possible), browser memory limits

**B) Server-Side Generation**:
- Send only data + columns to server action
- Server generates attachment (Node.js)
- Server sends directly to Microsoft Graph

**Pros**: No large client-to-server payload, cleaner separation  
**Cons**: Requires server-side PDF/Excel generation (different packages), code duplication

**C) Hybrid** (✅ **RECOMMENDED**):
- **For PDF/Excel**: Generate on client (existing code), send base64 to server
- **For CSV**: Lightweight, can generate on server directly from data
- Implement size check (reject if attachment > 10MB)

**Rationale**: 
- Most exports are small (< 1MB)
- Reuses existing export code (no duplication)
- Size limits prevent abuse
- If size becomes issue later, migrate to server-side generation

---

### Decision 2: Server Action vs API Route

**Options**:

**A) Server Action**:
```typescript
// src/server/actions/email.ts
export async function sendExportEmail(input) {
  "use server";
  // ...
}
```

**Pros**: Consistent with existing patterns, simple to call, built-in caching  
**Cons**: Cannot be called from external systems

**B) API Route**:
```typescript
// src/app/api/email/send/route.ts
export async function POST(request: Request) {
  // ...
}
```

**Pros**: RESTful, can be called externally  
**Cons**: Manual auth check, different error handling pattern

**✅ RECOMMENDATION**: **Server Action**

**Rationale**:
- Consistent with 100% of existing ERP server actions
- Email sending is internal-only operation (user-triggered from UI)
- Simpler error handling and type safety
- Future: If external API needed, wrap server action in API route

### Decision 3: Email Provider Abstraction

**Should we abstract email providers (Microsoft, SMTP, SendGrid) now, or focus on Microsoft only?**

**Option A**: Create provider interface now, implement Microsoft only:
```typescript
interface EmailProvider {

}

class MicrosoftGraphProvider implements EmailProvider { /* ... */ }
```

**Option B**: Implement Microsoft directly, refactor later if needed.

**✅ RECOMMENDATION**: **Option A - Provider abstraction now**

**Rationale**:
- Low cost (one interface + one implementation)
- Future-proofs for SMTP/SendGrid/Resend
- Cleaner testing (mock provider)
- Better separation of concerns

## 🔐 Security Considerations

### 1. Microsoft Credentials

**✅ MUST DO**:
- Store in `.env.local` (server-only, never `NEXT_PUBLIC_`)
- Never log credentials
- Never send to client
- Use `process.env` access from server only

### 2. Recipient Validation

**✅ MUST DO**:
- Validate email format (regex)
- Check for duplicates (dedupe To/CC/BCC)
- Enforce max recipient count (default: 20)
- Optional: Allow-list domains (future)

### 3. Attachment Size

**✅ MUST DO**:
- Max attachment size: 10 MB (default)
- Validate before sending
- Clear error message if exceeded

### 4. Rate Limiting

**⚠️ FUTURE**: Consider rate limiting email sends per user (e.g., 50/hour) to prevent abuse. Not required for Phase 002E.3 (internal admin users only).

### 5. Sensitive Data in Body

**✅ MUST DO**:
- Never log email body content to audit
- Log only metadata (to_count, subject, attachment_names)
- No base64 attachment in audit log
