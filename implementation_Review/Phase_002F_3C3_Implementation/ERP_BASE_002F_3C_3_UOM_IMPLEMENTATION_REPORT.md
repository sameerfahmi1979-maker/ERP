## 31. Known Limitations

### 1. uom_conversions Table Seeded Empty
The conversions table exists but contains no records. It is reserved for future special conversions. Normal same-category conversions use `conversion_factor_to_base` in `units_of_measure`.

### 2. No Operational Conversion Engine
The module provides master data storage only. No runtime conversion calculation API is implemented. Future modules must implement conversion logic using the `conversion_factor_to_base` values.

### 3. Packaging Conversions Are Item-Specific
Packaging units (BOX, BAG, etc.) have placeholder conversion factors of 1. Actual conversions (e.g., 1 BOX = 12 EA) are item-specific and must be defined in the Inventory Item Master module.

### 4. MONTH Is Approximate
The MONTH unit (730 hours) is an approximation. Exact monthly billing/rental logic (accounting for varying month lengths) must be handled in future billing/rental modules.

### 5. No UOM Validation Engine
No validation exists to prevent illogical operations (e.g., adding KG + M). Future modules must implement business logic to enforce unit compatibility.

### 6. Manual Browser QA Pending
Full browser testing was not performed in this implementation session.

### 7. Future Module Dependency
The UOM module is foundational. Its value will be realized when Inventory, Procurement, Fleet, Workshop, and other modules consume the UOM master data via the select components and server actions.

### 8. Export Functionality
Dedicated `exportX()` server actions were not implemented. ERPDataTable likely provides client-side export capabilities (as seen in Finance Basics), but server-side export actions were not created.

## 34. Post-Implementation Fix: Select Component Controlled State

**Date**: 2026-06-07 09:59 UTC+4

**Issue Identified**: 
Console error reported: "Base UI: A component is changing the uncontrolled value state of Select to be controlled."

**Root Cause**:
Both `UnitOfMeasureSelect` and `UomCategorySelect` components were rendering `<Select>` without a `value` prop during loading and error states, causing React to treat them as uncontrolled. When data loaded and the `value` prop was added, React detected the switch from uncontrolled to controlled state.

**Fix Applied**:
Modified both select components to always include the `value` prop, even in loading/error states:

```typescript
// Before (problematic)
if (loading) {
  return (
    <Select disabled={true}>
      <SelectTrigger className={className}>
        <SelectValue placeholder="Loading..." />
      </SelectTrigger>
    </Select>
  );
}

// After (fixed)
if (loading) {
  return (
    <Select value={value?.toString() ?? ""} disabled={true}>
      <SelectTrigger className={className}>
        <SelectValue placeholder="Loading..." />
      </SelectTrigger>
    </Select>
  );
}
```

**Files Modified**:
1. `src/components/erp/uom/unit-of-measure-select.tsx` (lines 125-145)
2. `src/components/erp/uom/uom-category-select.tsx` (lines 111-129)

**Verification**:
- TypeScript typecheck: ✅ PASS
- No new compilation errors introduced
- Select components now maintain controlled state throughout their lifecycle
