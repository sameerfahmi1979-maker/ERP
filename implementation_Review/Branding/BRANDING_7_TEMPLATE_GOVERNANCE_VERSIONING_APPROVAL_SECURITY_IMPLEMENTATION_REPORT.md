## 1. Executive Summary

BRANDING.7 delivers a complete template governance layer for the ALGT ERP report template system. It establishes lifecycle management (`draft → in_review → approved → published`), version lineage tracking (`parent_template_id`, `version_no`), a deterministic security review engine, an approval/rejection workflow with full audit trail, and a BRANDING.6 permission correction.

All formal ERP outputs and public QR issuance are now gated: only `approved` or `published` templates may be used via `createOutputPublicLink`. Templates in `draft`, `in_review`, `rejected`, or `archived` status are blocked from official public verification issuance at the server action layer.

Build result: **`npx tsc --noEmit` → exit 0 / `npm run build` → exit 0** ✅

## 5. Template Lifecycle Design

```
draft  ──→  in_review  ──→  approved  ──→  published
  ↑              │               │
  └── (rejected  ←──────────────┘
  resubmit)      │
                 ↓
             archived  (any status → archived)
```

- New templates start as `draft`
- `draft` and `rejected` templates can be submitted for review → `in_review`
- A security review runs automatically on submission; `failed` blocks the transition
- Approvers can `approve` or `reject` `in_review` templates
- Approved templates can be `published` by users with `reports.publish` or `reports.template.approve`
- Any non-archived template can be archived (sets `is_active = false`)
- `approved` / `published` templates can produce new draft versions via `createTemplateDraftVersion`

---

## 6. Versioning Behavior

- `version_no` already existed on the table (pre-BRANDING.7, defaulted to 1)
- `parent_template_id` is a nullable self-reference FK linking a draft version to its source
- `createTemplateDraftVersion(templateId)` copies all content fields, increments `version_no`, sets `governance_status = 'draft'`, `security_review_status = 'pending'`, `parent_template_id = templateId`
- The new template code gets a `_V{n}` suffix to maintain uniqueness
- The original `approved`/`published` template is never modified
- Logs `template_new_version_created` event on both the new template's event history

## 7. Approval / Publish Rules

| Action | Required Permission | Preconditions |
|---|---|---|
| Submit for Review | `reports.manage` | `governance_status = draft \| rejected` |
| Approve | `reports.template.approve` | `governance_status = in_review` + `security_review_status ≠ failed` |
| Reject | `reports.template.approve` | `governance_status = in_review` |
| Publish | `reports.publish` OR `reports.template.approve` | `governance_status = approved` |
| Archive | `reports.manage` OR `reports.template.approve` | not already archived |
| Create New Version | `reports.manage` | any non-archived/rejected status |

## 8. Security Review Rules

The security review (`runTemplateSecurityReview`) scans `body_html_en`, `body_html_ar`, `custom_css`, and `watermark_text` fields.

### Blocking patterns (prevent submission/approval)

| Rule | What it detects |
|---|---|
| `script_tag` | `<script` injection |
| `inline_event_handler` | `onclick=`, `onerror=` etc. (except safe `onerror="this.style.display='none'"`) |
| `raw_sql_*` | SELECT/INSERT/DELETE/UPDATE/DROP SQL fragments |
| `service_role_leak` | `service_role` string |
| `api_key_leak` | `api_key:` or `api-key=` patterns |
| `secret_leak` | `secret:` or `secret=` patterns |
| `token_leak` | Long token values in assignments |
| `salary_field` | salary/payroll/basic_pay/net_pay keywords |
| `iban_field` | iban/account_number keywords |
| `medical_field` | medical/diagnosis/prescription keywords |

### Warning patterns (noted but do not block)

| Rule | What it detects |
|---|---|
| `http_external_url` | Non-HTTPS external URLs |
| `ocr_extracted_text` | ocr_text / extracted_text references |
| `prompt_field` | prompt / system_prompt references |
| `embedding_field` | embedding / vector references |
| `sql_comment` | SQL comment pattern `-- word` |

The Executive Ledger renderer's known-safe `onerror="this.style.display='none'"` pattern is explicitly allowlisted.

## 9. QR Issuance Guardrails

`createOutputPublicLink` now enforces template governance when `template_id` is provided:

```typescript
if (input.template_id != null) {
  const templateCheck = await checkTemplateIsIssuable(input.template_id);
  if (!templateCheck.isIssuable) {
    return { success: false, error: `Cannot issue QR link: template is '${templateCheck.governanceStatus}'. Only approved or published templates may be used.` };
  }
}
```

- Passing no `template_id` (pilot letters without template context) is still allowed
- This prevents draft/in_review/rejected/archived templates from being used for official public outputs
- Internal preview of any template (via `resolveTemplatePreview`) is not blocked by governance status

## 13. Known Limitations

1. **Pilot letters (no template_id):** `LetterPreviewDialog` still issues QR links without a `template_id`. This is intentional for the pilot phase. BRANDING.8 should enforce template selection before QR issuance.
2. **No email notification on governance transitions:** Approvers are not notified when templates are submitted for review. Requires a notification system integration (out of scope for BRANDING.7).
3. **No `createTemplateDraftVersion` in UI:** The dropdown triggers `createTemplateDraftVersion` server-side and shows a toast with the new template ID, but does not navigate to or open the new draft. Full UX flow deferred to BRANDING.8.
4. **`governance_status` not enforced on `updateReportTemplate`:** The existing `updateReportTemplate` action does not check `governance_status`. Editing an approved/published template in place remains possible. BRANDING.8 should add the guard to prevent silent in-place mutation of approved templates.
5. **Security review warnings:** Warning-severity findings (HTTP URLs, prompt fields, OCR references) are recorded in `security_review_notes` but do not block the submission workflow. Reviewers must inspect findings manually.
