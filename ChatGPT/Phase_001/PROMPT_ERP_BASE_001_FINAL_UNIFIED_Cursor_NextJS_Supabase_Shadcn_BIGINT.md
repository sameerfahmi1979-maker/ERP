## 3. Critical Numbering Standard — No UUID Primary Keys for ERP Tables

This ERP must use normal numeric IDs, not UUID primary keys, for all ERP application/business tables.

### 3.1 Required ERP Table ID Format

All ERP application tables must use:

```sql
id bigint generated by default as identity primary key
```

This applies to all application tables including, but not limited to:

- owner_companies
- branches
- user_profiles
- roles
- permissions
- role_permissions
- user_roles
- audit_logs
- future HR tables
- future fleet tables
- future workshop tables
- future DMS tables
- future HSE tables
- future finance tables
- future procurement tables
- future inventory tables
- future diesel/weighbridge tables

### 3.2 Only Allowed UUID Exception

The only allowed UUID field is the Supabase Auth reference:

```sql
auth_user_id uuid references auth.users(id)
```

Reason:

Supabase Auth internally uses UUID for `auth.users.id`. This cannot be changed.

Rules:

- `user_profiles.id` must be BIGINT identity.
- `user_profiles.auth_user_id` may be UUID only because it links to Supabase Auth.
- No ERP business table may use UUID as its primary key.
- Do not generate UUID IDs for business records.
- Do not expose internal UUIDs in the UI except where technically unavoidable for Supabase Auth admin diagnostics.

Correct example:

```sql
create table public.user_profiles (
  id bigint generated by default as identity primary key,
  auth_user_id uuid not null unique references auth.users(id) on delete cascade,
  full_name text,
  display_name text,
  status text not null default 'active',
  created_at timestamptz not null default now(),
  updated_at timestamptz not null default now()
);
```

Incorrect example:

```sql
id uuid primary key default gen_random_uuid()
```

### 3.3 Human-Readable ERP Codes and Reference Numbers

Database primary key = BIGINT identity.

Business visible reference/code = human-readable text.

Use readable reference/code fields where needed.

Examples:

```sql
company_code text unique
branch_code text unique
user_code text unique
role_code text unique
permission_code text unique
```

Future transaction modules must use reference numbers such as:

```text
EMP-000001
VEH-000001
WBT-2026-000001
INV-2026-000001
PO-2026-000001
DMS-DOC-2026-000001
HSE-INC-2026-000001
```

Rules:

- Do not use UUIDs as visible record IDs.
- Do not expose UUIDs in list pages, forms, dashboards, or reports except Supabase Auth diagnostic pages if necessary.
- Every ERP table must be checked before final reporting.

### 3.4 Required Verification in Database Report

Before generating the final database report, scan all SQL migrations and TypeScript types.

Confirm explicitly in `ERP_BASE_001_DATABASE_REPORT.md`:

- No ERP table uses UUID as primary key.
- All ERP table primary keys are BIGINT identity.
- The only UUID usage is `auth_user_id` for Supabase Auth linking.
- All visible ERP records use numeric IDs and/or human-readable reference codes.
- No `gen_random_uuid()` is used for ERP business table primary keys.

If any UUID primary key is found in ERP application tables, stop and fix it before continuing.
