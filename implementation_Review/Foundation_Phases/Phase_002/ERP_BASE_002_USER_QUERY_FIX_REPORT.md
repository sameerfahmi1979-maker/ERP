## 2. Root Cause Analysis

### 2.1 Database Schema

The `user_roles` table has **two foreign key relationships** to the `user_profiles` table:

```sql
create table public.user_roles (
  id bigint generated by default as identity primary key,
  user_profile_id bigint not null references public.user_profiles(id) on delete cascade,
  role_id bigint not null references public.roles(id) on delete cascade,
  owner_company_id bigint references public.owner_companies(id) on delete cascade,
  branch_id bigint references public.branches(id) on delete cascade,
  is_active boolean not null default true,
  assigned_at timestamptz not null default now(),
  assigned_by bigint references public.user_profiles(id) on delete set null,  -- ⚠️ SECOND FK
  -- ...
);
```

**Two foreign keys**:
1. `user_profile_id` → `user_profiles(id)` - The user who has the role
2. `assigned_by` → `user_profiles(id)` - The user who assigned the role

### 2.2 Problematic Query

The original query attempted to use PostgREST's automatic relationship embedding:

```typescript
const { data: profiles, error } = await supabase
  .from("user_profiles")
  .select(`
    *,
    owner_companies:owner_company_id ( id, legal_name_en, company_code, status ),
    branches:branch_id ( id, branch_code, branch_name_en, status ),
    user_roles (                     // ⚠️ AMBIGUOUS: which FK to follow?
      is_active,
      roles ( role_code, role_name )
    )
  `)
  .order("id", { ascending: true });
```

### 2.3 PostgREST Relationship Inference Problem

When PostgREST encounters `user_roles` in the select for `user_profiles`, it doesn't know which foreign key path to follow:

- **Path A**: `user_profiles.id` ← `user_roles.user_profile_id` (the user's assigned roles)
- **Path B**: `user_profiles.id` ← `user_roles.assigned_by` (roles this user assigned to others)

Since both paths exist, PostgREST cannot automatically infer the relationship, resulting in the error.

## 3. Solution Strategy

### 3.1 Approach Selected

**Explicit Server-Side Query Split**

Instead of relying on PostgREST's automatic relationship embedding, we split the query into multiple explicit steps and merge the data on the server side.

### 3.2 Why This Approach?

✅ **No Migration Required**: Avoids database schema changes  
✅ **Preserves RLS**: All queries use the user-scoped Supabase client  
✅ **Full Control**: Explicit about which relationship to follow  
✅ **Type Safety**: Clear TypeScript types at each step  
✅ **Robust**: Continues gracefully if roles query fails  

### 3.3 Alternative Approaches Considered

**Option A: PostgreSQL View or RPC**
- Create a database view that explicitly resolves the relationship
- **Rejected**: Requires migration, added complexity, RLS review needed

**Option B: Explicit Relationship Naming in PostgREST**
- Use PostgREST hint syntax if available (e.g., `user_roles!user_profile_id()`)
- **Rejected**: Syntax not straightforward in Supabase JS client v2+

**Option C: Server-Side Split (Selected)**
- Fetch profiles, roles, and relationships separately
- Merge on server side
- **Selected**: Clean, no migration, full control

## 4. Implementation Details

### 4.1 New Query Structure

The fixed query follows a 5-step process:

#### Step 1: Fetch User Profiles with Company/Branch

```typescript
const { data: profiles, error: profilesError } = await supabase
  .from("user_profiles")
  .select(`
    *,
    owner_companies:owner_company_id ( id, legal_name_en, company_code, status ),
    branches:branch_id ( id, branch_code, branch_name_en, status )
  `)
  .order("id", { ascending: true });
```

**No ambiguity**: `owner_company_id` and `branch_id` are unambiguous foreign keys.

#### Step 2: Fetch User Roles Explicitly

```typescript
const profileIds = profiles.map((p) => p.id);
const { data: userRoles, error: rolesError } = await supabase
  .from("user_roles")
  .select(`
    user_profile_id,
    is_active,
    role_id,
    roles ( role_code, role_name )
  `)
  .in("user_profile_id", profileIds)  // ✅ EXPLICIT: follow user_profile_id FK
  .eq("is_active", true);
```

**Key**: The `.in("user_profile_id", profileIds)` filter explicitly tells PostgREST we want roles **assigned to** these users, not roles **assigned by** these users.

#### Step 3: Get Current User for Email

```typescript
const { data: { user: currentUser } } = await supabase.auth.getUser();
```

**Note**: Emails are only shown for the current user (RLS limitation for now).

#### Step 4: Build Role Map

```typescript
const rolesByProfile = new Map<number, { role_code: string; role_name: string }[]>();

if (userRoles) {
  for (const ur of userRoles) {
    const profileId = ur.user_profile_id;
    const roleData = ur.roles;
    
    if (roleData && typeof roleData === "object" && "role_code" in roleData && "role_name" in roleData) {
      if (!rolesByProfile.has(profileId)) {
        rolesByProfile.set(profileId, []);
      }
      rolesByProfile.get(profileId)!.push({
        role_code: String(roleData.role_code),
        role_name: String(roleData.role_name),
      });
    }
  }
}
```

**Map structure**: `profileId → role[]`

#### Step 5: Merge Data

```typescript
return profiles.map((profile) => {
  const roles = rolesByProfile.get(profile.id) ?? [];
  const { owner_companies, branches, ...rest } = profile as Record<string, unknown>;

  return {
    ...(rest as UserWithRoles),
    roles,
    owner_company: owner_companies as OwnerCompany | null,
    branch: branches as Branch | null,
    email: profile.auth_user_id === currentUser?.id ? currentUser?.email ?? null : null,
  };
});
```

**Result**: Clean `UserWithRoles[]` array with all data merged.

### 4.2 Error Handling

if (profilesError) {

## 9. Lessons Learned

### 9.1 PostgREST Relationship Ambiguity

**Problem**: Multiple foreign keys to the same table create ambiguity.

**Solution**: Use explicit filters or split queries.

**Best Practice**: When designing schemas with multiple FKs to the same table, consider:
1. Using explicit relationship names in PostgREST (if supported)
2. Creating views for common join patterns
3. Using server-side query composition

### 9.2 Server-Side Data Composition

**Benefit**: Full control over data shape and type safety.

**Trade-off**: Slightly more code, but much clearer intent.

**Recommendation**: For complex relationships, prefer explicit server-side composition over automatic ORM/PostgREST inference.

### 9.3 Graceful Degradation

**Pattern**: Continue with partial data rather than failing completely.

```typescript
if (rolesError) {
  console.error("listUsers user_roles error", rolesError.message);
  // Continue without roles rather than failing completely
}
```

**Result**: Users page shows profiles even if roles query fails (better UX).
