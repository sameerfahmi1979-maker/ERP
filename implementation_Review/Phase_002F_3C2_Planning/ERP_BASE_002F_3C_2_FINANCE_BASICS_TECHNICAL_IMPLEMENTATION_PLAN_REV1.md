## 1. Executive Summary

ERP BASE 002F.3C.2 delivers **Finance Basics / Commercial Readiness** master data: six dedicated tables (currencies, payment terms, tax types, banks, cost centers, profit centers) plus six global lookup categories. This phase prepares CRM, procurement, inventory, and future finance modules with reusable commercial reference data **without** implementing accounting, GL, budgets, exchange rates, or VAT filing.

**REV1 corrects v0** where it diverged from the live ERP architecture:

- **BIGINT** identity PKs and FKs (not `SERIAL`)
- Audit users via **`user_profiles.id` (BIGINT)** — not `auth.users.id` (UUID)
- **Actual RLS helpers** from `20260527120000_erp_base_foundation.sql` and geography migrations — not invented `auth_context()` / `has_permission()` patterns
- **One consolidated migration** — not 16 separate files
- **Removed accounting-like fields** (`exchange_rate_placeholder`, `budget_enabled`, `target_enabled`, `manager_user_id`)
- **Routes** under `/admin/master-data/finance-basics/*` — matching geography’s master-data grouping
- **Standing master-data reuse rule** for all future phases

**Scope remains approved:** currencies, payment terms, tax types, banks, cost centers, profit centers, and lookup categories PAYMENT_METHODS, TAX_TREATMENT_TYPES, BANK_ACCOUNT_TYPES, BANK_TYPES, COST_CENTER_TYPES, PROFIT_CENTER_TYPES.

**Estimated implementation effort:** 4–6 working days after Sameer approval (single migration + geography-parity feature module).

## 5. Corrected Lookup vs Dedicated Table Decision Matrix

| Entity | Storage | Table / Category | Rationale |
|--------|---------|------------------|-----------|
| Currencies | Dedicated table | `currencies` | Multi-field (ISO code, decimals, base flag), audit, lock/system governance |
| Payment Terms | Dedicated table | `payment_terms` | Structured commercial fields (due_days, advance %, retention %) |
| Tax Types | Dedicated table | `tax_types` | Rates, effective dates, VAT/RCM flags, sales/purchase applicability |
| Banks | Dedicated table | `banks` | `country_id` FK, SWIFT, contact fields |
| Cost Centers | Dedicated table | `cost_centers` | Hierarchy (`parent_cost_center_id`), org/branch scoping |
| Profit Centers | Dedicated table | `profit_centers` | Hierarchy (`parent_profit_center_id`), org/branch scoping |
| Payment Methods | Global lookup | `PAYMENT_METHODS` | Simple enumerated list (CASH, BANK_TRANSFER, CHEQUE, LC, etc.) |
| Tax Treatment Types | Global lookup | `TAX_TREATMENT_TYPES` | STANDARD, ZERO_RATED, EXEMPT, OUT_OF_SCOPE, REVERSE_CHARGE |
| Bank Account Types | Global lookup | `BANK_ACCOUNT_TYPES` | CURRENT, SAVINGS, ESCROW, etc. |
| Bank Types | Global lookup | `BANK_TYPES` | COMMERCIAL, ISLAMIC, CENTRAL, CORRESPONDENT |
| Cost Center Types | Global lookup | `COST_CENTER_TYPES` | DEPARTMENT, PROJECT, WAREHOUSE, etc. |
| Profit Center Types | Global lookup | `PROFIT_CENTER_TYPES` | BUSINESS_UNIT, PRODUCT_LINE, REGION, etc. |

**Rule:** Dedicated tables store `*_type_code` / `tax_treatment_code` / `bank_type_code` as TEXT referencing lookup `value_code` — not FK to lookup rows (matches geography port_type pattern).

**Dynamic lookup pages** (existing — no new pages):

- `/admin/master-data/lookups/categories`
- `/admin/master-data/lookups/values`
- `/admin/master-data/lookups/system`

Finance form dropdowns use `LookupSelect` with category codes; admin can add values without code deploy.

## 6. Corrected Database Schema Plan

### 6.1 Migration Strategy

**Single migration file:**

```text
supabase/migrations/YYYYMMDDHHMMSS_erp_base_002f3c2_finance_basics.sql
```

**Migration sections (in order):**

1. Create six tables + indexes + triggers
2. Enable RLS + create policies (24 policies: 4 × 6 tables)
3. Insert permissions + role assignments (DO $$ block)
4. Insert lookup categories + seed values (DO $$ block, `ON CONFLICT DO NOTHING`)

6. Optional: sample cost/profit center seeds (minimal, non-system)

Optional corrective migration only if post-QA fix required.

### 6.2 Table: `currencies`

```sql
create table if not exists public.currencies (
  id bigint generated by default as identity primary key,

  currency_code text not null unique,
  currency_name_en text not null,
  currency_name_ar text,
  symbol text,
  decimal_places integer not null default 2,

  is_base_currency boolean not null default false,

  description_en text,
  description_ar text,
  notes text,

  is_system boolean not null default false,
  is_locked boolean not null default false,
  is_active boolean not null default true,
  sort_order integer not null default 0,

  created_at timestamptz not null default now(),
  created_by bigint references public.user_profiles(id),
  updated_at timestamptz not null default now(),
  updated_by bigint references public.user_profiles(id),
  deactivated_at timestamptz,
  deactivated_by bigint references public.user_profiles(id),
  deactivation_reason text,

  check (currency_code = upper(currency_code)),
  check (currency_code ~ '^[A-Z]{3}$'),
  check (decimal_places between 0 and 4),
  check (
    (is_active = true and deactivated_at is null) or
    (is_active = false and deactivated_at is not null)
  )
);

create unique index if not exists idx_currencies_single_base
  on public.currencies (is_base_currency)
  where is_base_currency = true;

create index if not exists idx_currencies_code on public.currencies(currency_code);
create index if not exists idx_currencies_active on public.currencies(is_active);
create index if not exists idx_currencies_sort on public.currencies(sort_order);

create trigger trigger_currencies_updated_at
  before update on public.currencies
  for each row execute function public.set_updated_at();
```

**Notes:**

- No `exchange_rate_placeholder` — exchange rates are a future phase.
- Base currency enforced by partial unique index, not CHECK subquery.

## 10. Corrected RLS Policy Plan

### 10.1 Policy Template (All Six Tables)

Apply to: `currencies`, `payment_terms`, `tax_types`, `banks`, `cost_centers`, `profit_centers`.

**SELECT** — geography pattern:

```sql
create policy select_<table>_authenticated
  on public.<table>
  for select
  using (
    is_active = true
    or public.current_user_has_permission('master_data.finance_basics.view')
  );
```

**INSERT:**

```sql
create policy insert_<table>
  on public.<table>
  for insert
  with check (public.current_user_has_permission('master_data.finance_basics.manage'));
```

**UPDATE:**

```sql
create policy update_<table>
  on public.<table>
  for update
  using (
    public.current_user_has_permission('master_data.finance_basics.manage')
    and (
      not is_locked
      or public.current_user_has_role('system_admin')
    )
  );
```

**DELETE** — system_admin only (geography completion fix):

```sql
create policy delete_<table>
  on public.<table>
  for delete
  using (
    exists (
      select 1 from public.user_roles ur
      join public.roles r on r.id = ur.role_id
      where ur.user_profile_id = public.current_user_profile_id()
        and r.role_code = 'system_admin'
        and ur.is_active = true
    )
  );
```

### 10.2 Lock / Unlock Behavior

- `is_locked = true` records: only `system_admin` may UPDATE (via RLS UPDATE policy above).
- Lock/unlock actions in server actions check `hasPermission(ctx, "master_data.finance_basics.manage")` **and** `system_admin` role for lock toggle — mirrors geography `toggleCountryLock` pattern.
- System records (`is_system = true`): cannot hard delete (application layer rejects; optional DB trigger deferred).

### 10.3 Security Boundaries

- No public/anon write policies
- No bypass RLS for authenticated except via service role in migrations/seeds only
- Server actions always verify permissions before mutating
